## Что подготовлено

- Собрана новая продающая страница `/capping`; для Tilda она разложена на 6 HTML-блоков, потому что полный файл слишком большой для одного HTML-блока Tilda.
- Страница сделана по подходу `/excu`: hero, блок «Для кого», программа/формат, онлайн-запись, ценность каппинга, FAQ, финальный CTA и прямые контакты менеджера.
- Текущий виджет онлайн-записи не менялся: содержимое `schedule-online/capping-schedule-sync/tilda/capping-block.html` встроено в секцию «Онлайн-запись» как есть.
- Полный `tilda-block.html` оставлен как локальный источник/архив, но в Tilda его целиком не вставлять. Для Tilda использовать только файлы из `capping/tilda-blocks/` строго по порядку.
- Для расписания добавлена клиентская защита на случай проблем с доступом к API: последний успешный ответ хранится как fallback до 7 дней, а при полном отсутствии данных показывается понятный блок с листом ожидания, Telegram и WhatsApp.
- `booking_url` в расписании дополнительно проверяется на клиенте: используется только `https` и домены yClients; для schema.org `Event` небезопасная ссылка заменяется на страницу `/capping`.
- Добавлен SEO-слой: canonical/meta/OG/Twitter через JS и JSON-LD `WebPage`, `Service`, `FAQPage`, `BreadcrumbList`.
- FAQ сделан с плавным аккордеоном; вопросы актуализированы под открытые каппинги: добавлено объяснение формата для новичков, подготовка к дегустации и расширенный ответ про отдельный каппинг для команд.
- Добавлен плавный скролл к расписанию, который срабатывает только по существующим якорям и не должен ломать Tilda-хуки вроде `#waiting_list`.
- Hero-фото приведено к правилу дизайн-системы: фиксированная высота, без растяжения от высоты левой колонки. H1 сокращён до «Каппинг кофе», чтобы первый экран не раздувался.
- В hero добавлена кнопка «Оставить заявку» с ссылкой `#consalt` на Tilda-попап заявки по правилу `DESIGN_SYSTEM.md`. Не ставить `id="consalt"` на видимые элементы страницы.
- В hero оставлены две кнопки: «Выбрать дату» и «Оставить заявку». Кнопки «Задать вопрос» удалены из hero и финального CTA.
- В mobile hero «Оставить заявку» идёт верхней кнопкой на всю ширину, «Выбрать дату» остаётся нижней кнопкой на всю ширину.
- Остальные блоки дополнительно сверены с `DESIGN_SYSTEM.md`: порядок фонов, лейблы, карточки, mobile swipe для фото-карточек, FAQ, финальный CTA и контакты.
- Финальный CTA приведён к паттерну дизайн-системы: светло-зелёная карточка, сетка `1fr / 260px`, одна крупная кнопка «К расписанию» с inline SVG-иконкой и вторичные контакты менеджера под кнопкой.
- Контакты финального CTA актуализированы по `DESIGN_SYSTEM.md`: телефон, Telegram и WhatsApp с зелёными inline SVG-иконками, серым вторичным текстом и hover в зелёный.
- Для опубликованной Tilda добавлен runtime font guard в блок `05-page-scripts.html`: скрипт подключает Mulish в `<head>` и проставляет `font-family` inline с `important` на элементы `.mbs-capping-page` и `.mbs-capping`, потому что глобальные стили Tilda могут перебивать шрифт секций «Как проходит» и «Онлайн-запись».
- Дополнительно в блок `01-top-content.html` добавлен локальный font-fix рядом с секциями «Как проходит» и «Онлайн-запись»: он сразу проставляет Mulish inline и следит за изменениями `style/class`, если Tilda позднее возвращает свой шрифт.
- Мобильный hero приведён к порядку дизайн-системы: label + H1, фото, lead, факты, кнопки. Фото-карточки программы на mobile листаются горизонтально со скрытым scrollbar.
- В блоке «Как проходит» заменены 4 фото-карточки: «Знакомство с лотами», «Аромат и вкус», «Обсуждение», «Выводы».
- Блок «Зачем приходить» уплотнён по вёрстке: уменьшены вертикальные отступы, фото поднято и увеличено на desktop, карточки справа стали компактнее; для секции добавлен отдельный модификатор `mbs-capping-page__section--story`.
- Попап карточки онлайн-записи в виджете исправлен для локального iframe-preview: desktop-окно центрируется в текущей видимой области без скачка страницы, focus ставится с `preventScroll`, а scroll страницы блокируется вместе с родительским preview.
- Мобильный popup карточки онлайн-записи приведён ближе к рабочему исходному виджету: высота `85dvh`, контент прокручивается внутри `.mbs-cap-popup-scroll`, нижний CTA остаётся закреплённым.
- Для mobile убрана жёсткая блокировка scroll родительского preview/Tilda, потому что в iframe Safari она могла ломать свайп внутри карточки и давать пустую область над CTA.
- Деплой, Tilda API, backend, yClients, Google Sheets, cron и `.env` не менялись.

## Как проверить

1. Открыть `barista-course/capping/index.html` через VS Code Live Preview. Превью собирается из файлов `capping/tilda-blocks/`.
2. Проверить desktop и мобильную ширину.
3. Убедиться, что кнопка «Выбрать дату» скроллит к расписанию.
4. Проверить, что карточки расписания и popup виджета работают так же, как в исходном виджете.
5. Проверить FAQ-аккордеон.
6. Проверить контакты финального CTA: телефон, Telegram, WhatsApp.
7. После вставки в Tilda проверить `#waiting_list`, потому что сам лист ожидания остаётся Tilda-попапом на странице.

## Остаточные риски

- Встроенный виджет скопирован в страницу на момент сборки. Если потом отдельно обновлять `capping-block.html`, страницу `capping/tilda-block.html` нужно пересобрать или заново вставить актуальный виджет.
- Фото страницы используют Tilda CDN; если изображения удалят или заменят в Tilda, URL нужно обновить.
- Текущие атмосферные фото с Tilda CDN в основном портретные. Для hero это компенсируется фиксированной высотой и `object-fit: cover`; если появится горизонтальное фото каппинга, лучше заменить hero-изображение на него.
- Полный `tilda-block.html` слишком большой для одного HTML-блока Tilda. В Tilda вставлять только файлы из `capping/tilda-blocks/` по порядку `00 → 05`.
